# Route Archetypes

Proof-step phase 默认先在 4 种路线原型里做选择，而不是从零发明路线名词。

## 1. Vertical Slice

适合：
- 空项目
- MVP 第一刀
- 需要尽快跑通用户可感知的端到端路径

特征：
- 从输入到输出打一条最小纵切
- 功能可以少，但必须真的可运行、可验证
- 优先打通 happy path，再补失败路径与边角

常见危险：
- 顺手把未来模块一起搭骨架
- UI、后端、数据层一口气“都铺好”
- 没有 smoke / test / log 就说打通了

## 2. Seam-first

适合：
- 已有项目新增模块
- 需要沿现有边界插入新能力
- 需要先最小化对主系统的扰动

特征：
- 先找最清晰的接缝（seam）
- 优先沿已有模块、已有契约、已有测试入口下刀
- 先证明能稳定接入，再考虑抽象和扩展

常见危险：
- 没找 seam，直接改主干流程
- 接口边界不清，回归面过大
- 想一次把整块新模块全接进去

## 3. Harness-first

适合：
- 项目当前可观测性差
- 复现和定位问题困难
- 再加功能只会把已有混乱放大

特征：
- 先补 Run / Smoke / Logs / Tests / Repro / Fixtures
- 先让 agent 能看见真实反馈，再谈功能推进
- 先建立一条稳定 debug 路径

常见危险：
- 把 harness-first 变成“无限基建”
- 写很多工具和脚本，但没有服务于近期闭环
- 用“准备工作”逃避真正切第一刀

## 4. Transition Proof Step

适合：
- 文档、skill、架构说明需要进入当前 Goal Pack 角色
- 需要保留来源追溯，但不把来源材料当作当前执行入口
- 转换本身需要可验证的中间状态

特征：
- 先锁定输入材料和当前角色之间的映射
- 转换一个可验证的 Goal Pack 或 authority slice
- 每次转换写 evidence record，而不是写过程日记

常见危险：
- 同时保留多个 current home
- 把所有历史材料当 backlog
- 转换完成但没有 checker 或 review 证据

## 快速判断

如果你更关心：

- “先让人真的能用起来一点点” → `vertical-slice`
- “先安全接进现有系统” → `seam-first`
- “先让问题能复现、能验证、能定位” → `harness-first`
- “先把来源材料收进当前 Goal Pack 角色” → `transition-proof-step`

如果三者都有，优先问：**当前最妨碍 agent 稳定推进的障碍是什么？**

- 是没有明确切口？
- 是没有可感知价值线？
- 还是没有反馈回路？

那个障碍决定第一条路线。
